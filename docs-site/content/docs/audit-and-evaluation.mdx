## Audit Trail

The Audit Trail records workflow events such as:

- Scan start.
- Scan completion.
- Owner assignment.
- Finding assembly.
- Review support preparation.
- Human review decisions.
- Source registration or source-registration removal when supported.

Events should show structured context such as action, outcome, actor, object type, object ID, timestamp, previous state, resulting state, evidence references, policy-pack version, permission context, and review-support fingerprint when available.

Human-entered text should be sanitized before it appears in audit payloads or UI surfaces.

## Audit event fields

| Field | Why it matters | What to check |
| --- | --- | --- |
| Event type | Names the workflow action. | Scan, assignment, evidence assembly, and review events are represented separately. |
| Outcome | Shows accepted, completed, denied, or escalated state. | Denied or escalated actions remain visible instead of disappearing. |
| Actor | Identifies whether the system or a human acted. | Human review decisions have a human actor and required reason when accepted. |
| Object reference | Links the event to a scan, source, finding, or review. | References match the visible finding or source you are investigating. |
| Timestamp | Orders workflow events. | Review decisions happen after scan and evidence assembly events. |

## Finding timeline

Each finding detail page can show a scoped audit timeline. Use it to understand:

- When the finding was assembled.
- Which scan produced it.
- How owner assignment happened.
- Which review decisions were recorded.
- Whether a denial or simulated deletion boundary was recorded.

## Evaluation metrics

Evaluation helps the team and reviewers inspect scan quality. The surface can show:

<DocsScreenshot
  src="/docs/media/evaluation-metrics.png"
  alt="Evaluation page showing precision, recall, F1 score, reproducibility, throughput, paid-service cost, and reproducibility hashes."
  caption="Evaluation summarizes quality and resource intensity. It is evidence about the run, not a legal-compliance certificate."
/>

| Metric | What it means | Expected P0 interpretation |
| --- | --- | --- |
| Precision | Share of flagged items that were relevant. | Helps spot noisy evidence or overly broad rules. |
| Recall | Share of expected relevant items found. | Helps detect missed categories or unsupported file paths. |
| F1 score | Balance between precision and recall. | Useful for comparing scan versions or policy packs. |
| Reproducibility | Whether the same inputs produce stable results. | Deterministic runs should stay reproducible. |
| Throughput | Files processed per second. | Useful for capacity and demo readiness, not compliance. |
| Resource intensity | Memory, CPU, model calls, and paid-service cost. | Deterministic P0 runs should show zero model calls and zero paid-service cost. |
| Traceability hashes | Versions of dataset, rules, config, and evaluation. | Use them to compare whether two runs used the same basis. |

Evaluation metrics are product evidence, not a compliance certificate.

## Resource intensity

For deterministic P0 workflows, model calls and estimated paid-service cost should remain zero. Optional AI assistance, when configured, must be redacted-evidence-only, budget-guarded, and fail closed when safety checks cannot pass.

## What auditors should look for

- Every accepted review decision has an actor, reason, timestamp, and resulting status.
- Scan and review events refer to the correct source and finding IDs.
- Denied actions are visible rather than hidden.
- Evaluation explains quality and resource usage without claiming full GDPR compliance.
- Raw source content and unredacted personal data are not present in public payloads.
