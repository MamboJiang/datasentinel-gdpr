# Source Owner Assignment and Review Permissions

## Problem

Workspace membership is real, but Source ownership and review routing still have legacy demo behavior. A Source can be scanned without a Workspace member assignment, findings can be visible outside the assigned accountability path, and review transfer options can come from static governance demo data instead of active Workspace members.

## Options

1. Keep owner routing in governance mocks only.
   - Rejected because it does not use active Workspace membership and keeps the UI detached from the admin model.
2. Store a Source assignment against active Workspace member account IDs and route findings from that Source to that member.
   - Chosen because it is explicit, reversible, and matches the current Workspace boundary without adding external directory integration.
3. Add production directory owner attribution.
   - Rejected for P0 because production tenant sync is out of scope.

## State Machine

### Source Assignment

- `unassigned`: Source has no selected `assignedOwnerUserId`.
- `assigned`: Source has an active Workspace member account ID as its selected owner.
- `fallback_assigned`: Source has no selected owner, and the system routes findings to an active Data Steward member.

Events:

- `source_created(owner?)`: creates Source in `assigned` or `unassigned`.
- `source_owner_updated(owner?)`: changes Source assignment and updates current scan-derived finding ownership for that Source.
- `source_scanned`: produces findings owned by the selected owner, fallback Data Steward, or no owner.

Guards:

- Selected owners must be active members of the current Workspace.
- Workspace Admin permission can edit Source assignment, but Workspace Admin membership alone does not grant review decision authority.

Side effects:

- Assigned Source metadata stores owner display name and email copied from current Workspace membership.
- Findings derived from the Source inherit the owner object.
- Empty assignment does not expose findings to every Workspace member.

### Finding Visibility

- `hidden`: current actor is not the finding owner and does not have a DPO escalation role for the finding.
- `visible_assigned`: current actor is the finding owner or fallback steward.
- `visible_escalated`: current actor is in a DPO/Legal escalation path for an escalated finding.

Events:

- `list_findings`: returns only visible findings.
- `get_finding`: returns 404 for hidden findings.
- `review_support_requested`: returns decision options only for visible findings.

### Review Decision

- `assigned`: accountable owner can record supported decisions.
- `delete_candidate`: owner confirmed that deletion is only marked, not executed.
- `retained`: owner recorded a retention exception with a reason and review date.
- `false_positive`: owner recorded a correction.
- `assigned`: owner transferred accountability to another active Workspace member.
- `escalated`: owner escalated to DPO or Legal.

Guards:

- `delete_candidate` requires an explicit confirmation checklist item.
- `reassign_owner` target must be an active Workspace member in current review support options.
- Real deletion remains denied.

## Impact Surface

- API contract: Source may include `assignedOwnerUserId`, `assignedOwner`, and `fallbackOwner`; Source supports owner update.
- Backend: Source create/update validation, current Workspace member lookup, source/finding visibility, review support generation, review validation.
- Frontend: Source create/edit owner controls, Source table owner display, review transfer options, deletion-candidate confirmation.
- Persistence: Source assignment is stored in the existing JSON source record.

## Rollback

- Remove Source assignment fields from new Source records or set them to `null`.
- Revert Source PATCH route and frontend edit controls.
- Findings from a scan can be regenerated by rescanning the Source after rollback.

## Acceptance

- Source creation can select an active Workspace member as owner or leave owner empty.
- Admin Sources can edit the Source owner.
- Non-admin users do not see Source findings unless the finding owner is their account or a supported fallback/escalation route.
- Workspace Admin group membership alone does not grant business review decisions.
- Review transfer targets come from active Workspace members, not static demo governance data.
- Marking a deletion candidate requires explicit confirmation and still reports `deletionExecuted = false`.
